# Migracja RouteMarket z Lovable Cloud na własny VPS

Dokument opisuje **wszystko**, czego potrzebujesz, aby odciąć projekt od Lovable Cloud i utrzymywać go samodzielnie na własnym VPS, z repozytorium na GitHub.

---

## 1. Aktualna architektura (stan przed migracją)

**Frontend** — Vite + React 18 + Tailwind, hostowany przez Lovable.

**Backend = Lovable Cloud** (cienka warstwa nad Supabase). Konkretnie używamy:

| Warstwa | Co | Gdzie żyje |
|---|---|---|
| Postgres + RLS | 28+ migracji w `supabase/migrations/` | Lovable-managed Supabase |
| Auth | email/hasło + Google OAuth (przez `@lovable.dev/cloud-auth-js`) | Lovable Auth proxy → Supabase Auth |
| Storage | buckety: `route-covers`, `gpx-files`, `pdf-guides`, `poi-images`, `marketing-assets` | Supabase Storage |
| Edge Functions (Deno) | 19 funkcji w `supabase/functions/` | Supabase Edge Runtime |
| Realtime | postgres_changes na `messages` | Supabase Realtime |
| AI Gateway | `LOVABLE_API_KEY` → Gemini / GPT-5 | **Lovable-only** |
| Email | `process-email-queue` + SMTP `notify.routemarket.io` | własny SMTP, kolejka pgmq w Postgresie |
| Płatności | Stripe + Stripe Connect | własne konto Stripe |

**Integracje zewnętrzne nadal Twoje (bez zmian):** Stripe, OpenStreetMap (Leaflet — bez API key), domena `routemarket.io`, SMTP `notify.routemarket.io`.

| 7 | `lovable-tagger` plugin Vite (`vite.config.ts`) | NIE — tylko dev | Można usunąć całkowicie. Już dziś włącza się tylko w `mode === 'development'`. |
| 8 | Plik `src/integrations/supabase/client.ts` „auto-generated by Lovable” | NIE | Po migracji edytuj go ręcznie — komentarz „do not edit” przestaje obowiązywać. |
| 9 | `src/integrations/supabase/types.ts` (auto-gen) | NIE | Generuj lokalnie: `npx supabase gen types typescript --local > src/integrations/supabase/types.ts`. |

| 11 | Cron / scheduled jobs (jeśli aktywne na Lovable) | częściowo | `process-email-queue` był odpalany cyklicznie — na VPS użyj `pg_cron` (już dostępny w Supabase) lub systemd timer wywołujący endpoint. |
| 12 | Domeny (`*.lovable.app`) | NIE | `routemarket.io` jest Twoje. Wskaż A-record na IP VPS. |
| 13 | Storage signed URLs (5 min) | NIE | Mechanizm działa identycznie na self-hosted Supabase Storage. |
| 14 | Realtime (`messages`) | NIE | Self-hosted Supabase ma Realtime out of the box. |
| 15 | MCP server (`mcp-server` edge function) | NIE | Działa wszędzie. Zaktualizuj URL w klientach MCP. |

## 3. Lista ryzyk

1. **Auth migration** — użytkownicy logujący się przez Google przez `lovable-auth` mogą musieć ponownie się zalogować (zmiana issuera JWT). Przy „praktycznie zero użytkowników” akceptowalne.
2. **Eksport Storage** — pliki w bucketach trzeba ściągnąć (`supabase storage download` lub przez panel) i wgrać do nowego bucketa. Zachowaj **klucze** (paths) 1:1 — w bazie trzymamy `cover_image_key`, `gpx_file_key` itd.
3. **JWT secret** — jeśli zmienisz `JWT_SECRET`, wszystkie istniejące tokeny stają się nieważne. To jest ok przy migracji, ale zaplanuj „logout all users”.
4. **Edge runtime różnice** — self-hosted edge runtime czasem wolniej dostaje aktualizacje API niż Lovable. Funkcje używają standardowego Deno + `@supabase/supabase-js` i nie powinny niczego specyficznego potrzebować.
5. **Lovable AI Gateway → bezpośredni provider** — koszty mogą się zmienić; trzeba wziąć osobne klucze i utrzymać quoty.
6. **DNS / SSL** — przy przełączeniu DNS przewidź ~1h propagacji. Caddy pobierze cert Let's Encrypt automatycznie.
7. **Sekrety w repo** — `.env` MUSI być w `.gitignore`. Repo eksportowane z Lovable może już mieć `.env` z prawdziwymi kluczami — przed pierwszym pushem wyczyść historię.

## 4. Kolejność migracji — krok po kroku

### Faza A — przygotowanie (lokalnie, bez ruchu produkcyjnego)
1. **Eksport repo**: GitHub → Lovable → „Connect to GitHub” → push. Klonuj lokalnie.
2. **Wyczyść `.env`** z prawdziwych sekretów. Sprawdź `.gitignore`.
3. **Postaw Supabase lokalnie**: `git clone https://github.com/supabase/supabase`, `cd supabase/docker`, `cp .env.example .env`, ustaw hasła, `docker compose up -d`. Panel: `http://localhost:8000`.
4. **Wgraj migracje**: zainstaluj Supabase CLI, `supabase link --project-ref local --db-url $SUPABASE_DB_URL`, `supabase db push` (lub `psql $SUPABASE_DB_URL -f supabase/migrations/*.sql`).
5. **Stwórz buckety** zgodne z tabelą w §1 (public/private flagi).
6. **Skonfiguruj Auth**: w panelu Supabase włącz email + Google provider (`GOOGLE_CLIENT_ID/SECRET` z Google Cloud Console). Site URL = `http://localhost:5173` (na razie).
7. **Frontend**: `cp .env.example .env`, ustaw `VITE_SUPABASE_URL=http://localhost:8000`, `VITE_SUPABASE_PUBLISHABLE_KEY=` z lokalnego stacku. `npm install && npm run dev`. Sprawdź logowanie + utworzenie trasy.
8. **Edge functions**: `supabase functions deploy --no-verify-jwt` dla każdej. Dla AI funkcji **najpierw** podmień `LOVABLE_API_KEY` na bezpośrednie klucze providerów (sekcja §5).

### Faza B — VPS (staging)
9. Załóż VPS (Hetzner/DO/Vultr, min. 4 GB RAM dla pełnego Supabase). Ubuntu 24.04.
10. Zainstaluj Docker + Docker Compose. Otwórz porty 80, 443.
11. Sklonuj repo, sklonuj `supabase/docker` w katalogu obok (`./supabase-self-hosted/`). Skonfiguruj `.env` w obu.
12. Wystartuj Supabase: `docker compose -f supabase-self-hosted/docker-compose.yml up -d`.
13. Wystartuj frontend + Caddy: `docker compose up -d --build`.
14. Testuj na sub-domenie `staging.routemarket.io` (zanim przełączysz produkcyjną).

### Faza C — przeniesienie danych (jeśli chcesz zachować istniejące trasy/kontent)
15. Eksport bazy z Lovable: poproś support Lovable o dump `pg_dump`, lub w panelu Cloud → Database → Export. Wgraj na nowego Postgresa: `psql $NEW_DB_URL < dump.sql`.
16. Eksport Storage: dla każdego bucketu pobierz listę obiektów i przegraj (skrypt w §7).
17. Re-issue klucze Stripe webhook (nowy URL `https://your-domain.tld/functions/v1/stripe-webhook`).

### Faza D — cutover
18. Przełącz DNS `routemarket.io` na IP VPS.
19. W panelu Supabase ustaw `Site URL = https://routemarket.io` + `Additional Redirect URLs`.
20. Zaktualizuj URL Stripe webhook w dashboardzie Stripe.
21. Wyłącz projekt w Lovable Cloud (po 7-14 dniach obserwacji).

## 5. Co robić z `LOVABLE_API_KEY` (AI funkcje)

Funkcje używające `https://ai.gateway.lovable.dev/v1/chat/completions`:
- `generate-ai-suggestion`
- `content-generator`
- `generate-content-package`
- `auto-translate-route`
- `translate-route`
- `generate-pdf` (sprawdź)

Ten URL jest **OpenAI-kompatybilny**, więc wymiana to:
- **OpenAI**: `https://api.openai.com/v1/chat/completions`, klucz `OPENAI_API_KEY`, model `gpt-4o-mini` / `gpt-5-mini`.
- **Google Gemini**: użyj OpenAI compatibility layer `https://generativelanguage.googleapis.com/v1beta/openai/chat/completions` z `GOOGLE_AI_API_KEY`.

Zmień stałą `AI_GATEWAY_URL` i nazwy modeli w 5 plikach. Reszta kodu zostaje.

## 6. Co musisz zrobić ręcznie (Lovable nie zrobi tego za Ciebie)

- [ ] **Push do GitHub** — w panelu Lovable: Settings → GitHub → Connect → wybierz/utwórz repo, push.
- [ ] **Wyeksportować sekrety** z panelu Lovable Cloud → Secrets. Lista do skopiowania: `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `LOVABLE_API_KEY` (lub nowe `OPENAI_API_KEY`/`GOOGLE_AI_API_KEY`), `API_READONLY_KEY`, `MCP_CREATOR_USER_ID`, dane SMTP. Wartości tych sekretów **nie są widoczne** dla agenta — musisz je przepisać samodzielnie.
- [ ] **Eksport bazy** — Lovable Cloud → Database → Export (lub poproś support o `pg_dump`).
- [ ] **Eksport Storage** — pobierz pliki ze wszystkich 5 bucketów. Lovable nie udostępnia bulk-download w UI.
- [ ] **Google OAuth** — utwórz nowe credentials w [Google Cloud Console](https://console.cloud.google.com/apis/credentials) z redirect URI `https://supabase.your-domain.tld/auth/v1/callback`.
- [ ] **Stripe** — w dashboardzie Stripe podmień URL webhook na nowy.
- [ ] **DNS** — A-record `routemarket.io` → IP VPS, A-record `supabase.routemarket.io` → IP VPS.
- [ ] **SMTP** — wybierz dostawcę (Resend / Postmark / SES) i wgraj nowe credentials do `.env`.

## 7. Pomocniczy skrypt eksportu Storage

```bash
# wymaga Supabase CLI i SUPABASE_SERVICE_ROLE_KEY w env
for bucket in route-covers gpx-files pdf-guides poi-images marketing-assets; do
  mkdir -p "./storage-dump/$bucket"
  supabase storage cp "ss:///$bucket/" "./storage-dump/$bucket/" --recursive
done
# upload na nowy projekt:
for bucket in route-covers gpx-files pdf-guides poi-images marketing-assets; do
  supabase storage cp "./storage-dump/$bucket/" "ss:///$bucket/" --recursive
done
```

## 8. Uruchomienie projektu poza Lovable (TL;DR)

```bash
git clone git@github.com:<you>/routemarket.git
cd routemarket
cp .env.example .env       # uzupełnij wartości

# Wariant A: tylko frontend, Supabase już stoi gdzie indziej
npm install
npm run dev                # http://localhost:5173

# Wariant B: wszystko w Dockerze na VPS
docker compose up -d --build
# → frontend na :80/:443 przez Caddy, Supabase oddzielnie (patrz DEPLOY_VPS.md)
```
